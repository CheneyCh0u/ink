# Finder 文件夹拖入项目设计

## 背景与目标

Ink 已支持通过目录选择面板添加项目，也支持侧边栏内部拖动排序，但 Finder 文件夹
拖到侧边栏时不会产生项目。roadmap 的 P1-A 多项目工作流要求“Finder 文件夹可拖入
侧边栏添加为项目；侧边栏内部拖动继续只负责排序”。

本项要让展开态和图标态侧边栏都接受 Finder 本地目录，支持一次拖入多个目录，同时
保持现有置顶分组、内部排序、项目持久化和按需启动 shell 的边界。改动只发生在
`InkShell`，不进入 TerminalCore、PTY、Metal、grid 或渲染热路径。

## 范围

### 本次实现

- Finder 单个或多个本地目录可拖入整个可见侧边栏。
- 多目录按 Finder pasteboard 顺序批量添加。
- 普通文件、非本地 URL、失效目录、同批重复目录和已有项目被安全过滤。
- 新项目保持未置顶，追加到普通项目区末尾；置顶项目顺序不变。
- 含新项目时选中第一个新项目；全是重复项目时选中第一个已有项目。
- “新建项目”目录面板与 Finder 拖入复用同一批量导入决策。
- 内部项目拖动仍只排序，继续受置顶区 / 普通区边界约束。
- 使用系统 `.copy` / `.move` 拖放光标，不增加自绘覆盖层、动画或提示条。

### 明确不做

- 不接受普通文件并推导其父目录。
- 不递归扫描拖入目录，也不把目录子项变成项目。
- 不解析文件别名或符号链接到真实路径；用户拖入的符号链接目录保留该路径身份。
- 不增加安全作用域书签、项目导入历史、撤销栈或拖放设置项。
- 不改变项目置顶、备注、标签色、删除确认或会话恢复格式。
- 不为无效拖入弹窗；系统拒绝光标已经表达结果。

## 方案比较

### 方案 A：侧边栏解码拖放，窗口控制器应用导入计划（采用）

侧边栏只识别内部拖动与 Finder file URL，Foundation 导入规划器负责目录校验、
标准化和去重，窗口控制器一次性修改项目状态。这保留了 AppKit 拖放、纯决策和应用
状态三层边界，容易分别测试，也能让目录面板复用同一规则。

### 方案 B：侧边栏直接创建并保存项目

文件集中但会让视图知道 `Project`、UserDefaults、活动项目和 shell 启动，破坏现有
控制器持有工作区状态的分层，因此不采用。

### 方案 C：引入通用拖放服务

可以覆盖未来文件、标签和命令拖入，但当前只有项目目录一个场景。额外服务和协议没有
实际消费者，属于提前抽象，因此不采用。

## 组件与边界

### `ProjectDirectoryImportPlanner`

新增 Foundation-only 的值类型规划器，放在 `InkShell`。输入是候选 URL 和当前项目
目录，输出 `ProjectDirectoryImportPlan`：

- `directoriesToAdd: [URL]`：按输入顺序保留的全新目录；
- `selectedIndex: Int?`：应用追加后应选中的项目索引。

规划器同时提供不依赖现有项目列表的 `validDirectories(from:)`，供侧边栏判断系统拖放
光标；最终 `plan(candidates:existingDirectories:)` 会再次调用同一校验，避免悬停后目录
状态变化造成 TOCTOU 错误。

规划器不创建 `Project`、不访问 UserDefaults、不启动 shell，也不依赖 AppKit。它负责：

1. 只保留 `isFileURL == true`、当前存在且 `isDirectory == true` 的 URL；
2. 用 `standardizedFileURL.path` 作为本次导入与已有项目的身份；
3. 保留候选顺序，去掉同批重复；
4. 收集全新目录，并记住第一个新目录追加后的索引；
5. 若没有新目录，退回第一个命中的已有项目索引；
6. 若没有任何有效或已有目录，返回空计划且 `selectedIndex == nil`。

`standardizedFileURL` 会消除 `.`、`..` 和多余路径分隔，不解析符号链接。这样项目显示
继续忠实于用户选择的路径，也与现有持久化字符串语义一致。Finder 和目录面板通常给出
文件系统规范大小写；本项不引入资源标识符或真实路径缓存。

目录存在性在拖入悬停和最终应用之间可能变化。侧边栏只把悬停判断当作光标提示，窗口
控制器在执行时重新生成计划；目录已消失时安全退化为空操作。

### Finder pasteboard 解码

侧边栏根视图继续承接所有拖放。它同时注册：

- `SidebarViewController.dragType`：Ink 内部项目行；
- `.fileURL`：Finder 本地 file URL。

解码顺序固定为内部类型优先：

1. 若有合法内部行索引，返回内部排序意图；
2. 否则通过 `NSPasteboard.readObjects(forClasses:options:)` 按 pasteboard 顺序读取本地
   file URL；
3. `validDirectories(from:)` 至少得到一个目录时返回外部导入意图；
4. 其余载荷拒绝。

内部拖动在 `draggingEntered` / `draggingUpdated` 返回 `.move`，外部目录返回 `.copy`，
拒绝返回 `[]`。`performDragOperation` 对内部意图调用现有 `onReorder`；对外部意图调用
新增 `onImportDirectories([URL])`。外部拖入不使用落点决定项目位置，因为新项目的范围
规则明确为普通区末尾。

根视图覆盖侧边栏完整区域，因此展开态、图标态、项目行空白、标题和底部入口上方都
使用同一拖入行为。侧边栏完全隐藏时没有可见目标，不额外建立窗口级 drop zone。

### `MainWindowController` 批量应用

窗口控制器新增单一 `importProjectDirectories(_:)` 入口，目录面板和侧边栏回调都调用它：

1. 用当前 `projects.map(\.directory)` 生成导入计划；
2. 将 `directoriesToAdd` 映射成默认未置顶、无备注、无标签色的 `Project` 并一次追加；
3. 若计划有 `selectedIndex`，只调用一次项目选择路径；
4. 项目选择负责一次项目持久化、活动项目记录、侧边栏 / 标签栏刷新和工作区保存调度；
5. 只有被选中的空项目按现有规则创建首个标签与 shell，其余新项目保持空标签，首次
   选中时才启动。

规划阶段不逐项修改数组，应用阶段也不逐项调用 `persistProjects()` 或
`selectProject(at:)`。因此批量拖入不会按目录数量放大 UserDefaults 写入、窗口重建或
PTY 数量。现有保存调度器仍可合并同一主循环中的工作区状态变化。

若首个新项目启动 shell 失败，项目本身仍保留并保持选中，与目录面板当前的空项目
容错一致；用户之后再次选择或新建标签即可重试。

## 数据流

```text
Finder / 项目行
      │
      ▼
ProjectDropView ──内部类型──► onReorder(from, to)
      │
      └──file URLs──► onImportDirectories(urls)
                              │
                              ▼
                 ProjectDirectoryImportPlanner
                              │
                              ▼
                 append Project[] + select once
                              │
                              ▼
                 ProjectStore / workspace / chrome
```

## 错误与边界处理

- pasteboard 无法解码、URL 不是本地文件、目标不是目录或读取时已消失：忽略该项。
- 一批载荷没有任何有效目录或已有项目：返回 `false`，不修改状态、不弹窗。
- 同批重复：保留第一次出现的位置，后续忽略。
- 与已有项目重复：不创建第二个 `Project`；全重复时切换到第一个匹配项目。
- 混合“已有 + 新目录”：即使已有项目先出现，也选中第一个新项目，确保新增动作有可见
  结果；候选顺序只决定新增项目之间的顺序。
- 内部自定义类型和 `.fileURL` 同时存在：内部类型优先，避免拖动项目行意外走导入。
- 置顶项目存在时：新项目追加在数组尾部，置顶区和普通区现有相对顺序均不改变。

## 测试与验收

### 导入规划器

- 多目录保持输入顺序，同批重复只保留第一次。
- `.` / `..` 等标准化等价路径与已有项目去重。
- 普通文件、非 file URL 和失效目录被过滤。
- 混合已有与新目录时新增全部新目录并选择第一个新目录。
- 全重复时选择第一个已有项目；全无效时得到空计划。
- 新目录索引位于现有数组末尾，不改变置顶项目顺序。
- 符号链接目录保留符号链接 URL，不解析真实路径。

### 侧边栏拖放

- 内部项目 pasteboard 优先并返回 `.move`。
- Finder 有效目录返回 `.copy`，普通文件载荷拒绝。
- 多个 file URL 按 pasteboard 顺序交给导入回调。
- 展开态与图标态复用同一个根 drop view 和回调。
- 现有内部拖动落点、置顶区夹取和排序测试继续通过。

### 窗口与持久化

- 批量添加后项目顺序、默认元数据和活动索引正确。
- 只为第一个新项目创建首个 pane；其它新增项目保持空标签。
- 全重复时不增加项目并切换到已有项目。
- 目录面板使用同一入口，不再维护独立去重分支。
- 项目 JSON、活动项目路径和工作区快照可往返。

### 人工验证

1. 从 Finder 向展开态侧边栏拖入一个目录，看到 copy 光标并新增、选中项目。
2. 一次拖入多个目录，确认顺序与 Finder 一致且只有第一个新项目启动 shell。
3. 混合目录、普通文件、重复目录，确认只添加有效新目录。
4. 切到图标态重复拖入，行为一致且布局不变化。
5. 拖动已有项目行，确认仍显示 move 光标并按原规则排序。

回归门禁为 `swift test`、`swift build` 和 `git diff --check`。本项不改热路径，不需要
Time Profiler；不得改变 `Cell`、`RowInfo` 或任何 scrollback 常驻结构。
