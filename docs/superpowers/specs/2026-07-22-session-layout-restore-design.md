# 会话布局恢复设计

Issue：[#68](https://github.com/CheneyCh0u/ink/issues/68)

## 背景

Ink 当前用 `ProjectStore` 持久化项目目录、置顶、备注、颜色和最后活动项目，但标签与
pane 都是纯运行态。应用重启后，每个项目的标签名称、工作目录、活动位置、递归分屏
结构与比例都会丢失，用户需要手动重建工作区。

本功能只恢复工作区的“形状”和启动目录。旧 PTY、前台进程、命令、环境变量、终端
内容和 scrollback 都不具备安全且可靠的跨进程恢复语义，不能进入快照。恢复出的每个
pane 必须启动一个全新的 shell。

## 目标与范围

- 在重启后恢复项目对应的标签顺序、自定义名称与活动标签。
- 恢复每个标签的递归分屏结构、归一化比例与活动 pane。
- 记录各 pane 最近可取得的工作目录，并从该目录启动新 shell。
- 缺失目录、损坏字段或局部无效布局只影响对应项目、标签或 pane，不阻塞 Ink 启动。
- 工作区变化合并保存，正常关闭窗口或退出应用前强制保存最新快照。
- 编解码、保存调度和目录查询不得进入终端输出、parser、grid、scrollback 或渲染热路径。

本次不恢复进程、PTY、终端尺寸、标题栏的 OSC 动态标题、命令状态、通知、搜索状态、
选区、viewport 位置或 scrollback。命令状态与通知仍按 roadmap 作为后续独立功能。

## 方案选择

采用独立的、版本化 `WorkspaceSnapshot`，编码为单个 JSON `Data` 存入
`UserDefaults`。现有 `ProjectStore` 继续负责稳定的项目列表与元数据，工作区快照按
标准化项目路径与其关联。两者职责分开：修改项目备注或颜色不需要理解会话树，标签和
分屏的高频状态也不改变现有项目格式。

没有选择把会话字段直接加入 `ProjectStore.Stored`，因为这会把项目元数据迁移与运行态
恢复绑在一起，并让任一标签损坏导致整个项目列表无法解码。没有选择 Application
Support 下的独立文件，因为快照很小、只供 Ink 自身读取，文件协调、目录创建与原子替换
不会带来相称收益。`UserDefaults` 中的单值 `Data` 也便于测试使用独立 suite 隔离。

## 数据模型

`WorkspaceStore` 位于 `InkShell`，注入 `UserDefaults`，固定 key 为
`ink.workspace.v1`。外层快照包含：

```json
{
  "schemaVersion": 1,
  "activeProjectPath": "~/work/code/ink",
  "projects": [
    {
      "path": "~/work/code/ink",
      "activeTabIndex": 0,
      "tabs": [
        {
          "customName": "开发",
          "activePaneID": "B4E8…",
          "layout": {
            "kind": "group",
            "axis": "leftRight",
            "weights": [0.62, 0.38],
            "children": [
              {
                "kind": "leaf",
                "paneID": "B4E8…",
                "workingDirectory": "~/work/code/ink"
              },
              {
                "kind": "leaf",
                "paneID": "8D47…",
                "workingDirectory": "~/work/code/ink/Tests"
              }
            ]
          }
        }
      ]
    }
  ]
}
```

`schemaVersion` 必须等于 1。更高版本明确拒绝，损坏 JSON 返回无快照。项目路径和工作
目录保存为带 `~` 的绝对路径，读取后展开并标准化。UUID 只用于一次快照内连接布局叶子
和活动 pane；恢复后生成新的运行时 `PaneID` / `SplitID`，不把持久化标识当进程身份。

布局 DTO 是递归的叶子/分组枚举：

- 叶子包含持久化 pane ID 与工作目录；
- 分组包含轴向、权重与至少两个子节点；
- `weights.count` 必须等于 `children.count`，所有权重必须有限且大于零；
- 合法权重读取后重新归一化，分组使用新的 `SplitID`；
- pane ID 必须非空且在标签内唯一，活动 ID 不存在时回退到布局第一个叶子。

快照不复制 `ProjectStore` 的 `pinned`、`note` 或 `label`。恢复时先加载项目，再按标准化
路径关联快照。快照中已不在项目列表的记录忽略；项目列表中没有快照的项目保留为空，
第一次选中时沿用现有行为创建一个新标签。没有工作区快照的旧版本因此自然迁移，不需要
改写旧 key。

## 校验与资源上限

快照属于本机数据，但仍可能被旧版本、测试或手工修改损坏。恢复器使用明确上限，避免
启动时制造无界递归或 PTY 风暴：

- 最多读取 64 个项目记录；
- 每项目最多恢复 32 个标签；
- 每标签最多 16 个 pane，布局深度最多 16；
- 全部项目合计最多启动 128 个 pane。

项目或标签超过上限时不截断递归树，而是跳过超预算的后续标签；单个无效标签只被丢弃，
其它合法标签继续恢复。活动索引统一夹回合法范围。一个项目的所有标签都无效时保持空
项目，选中后由现有首标签逻辑兜底。上限只约束从不可信快照恢复，不改变用户在当前运行
期的操作能力。

## 目录恢复策略

`TerminalSession` 暴露只读的快照目录：优先使用当前前台进程目录，查询失败则使用创建
会话时的初始目录。生成快照时才执行前台目录查询；终端每次输出和绘制都不查询。

读取时：

1. `ProjectStore` 已有规则继续静默剔除不存在或不是目录的项目，包括断开的外置盘。
2. pane 保存目录存在且是目录时使用该目录。
3. pane 目录失效时回退到所属项目目录。
4. 项目目录也不可用时回退到用户主目录。

恢复单个 pane 启动失败时丢弃该标签的已启动 pane，并继续恢复其它标签；不会留下半棵
布局树。若所有恢复均失败，窗口沿用当前默认项目与新会话行为，并只显示已有启动错误，
不因快照再弹额外错误。

## 模型与恢复边界

`WorkspaceStore` 只处理 Codable DTO 和 `UserDefaults` 读写，不依赖 AppKit、Metal、PTY
或 `TerminalCore`。它提供纯数据的 `load()` / `save(_:)`，并允许注入独立 defaults suite，
便于测试且不会污染用户状态。`MainWindowController` 同样接收该 store；生产初始化使用标准
defaults，测试初始化使用隔离实例。

`TerminalTab` 增加受控恢复入口，一次接收已经校验且完整的 `PaneLayout`、pane 字典、
活动 pane 与自定义名称。构造器再次断言布局叶子集合与字典键集合一致，避免持久化 DTO
绕过运行态不变式。普通分屏、关闭和激活操作继续使用现有方法。

`MainWindowController` 负责把 DTO 映射为运行态，因为只有外壳知道项目列表、配置中的
scrollback 容量、PTY 启动及失败提示。初始化顺序为：

1. 加载 `ProjectStore`，必要时建立默认 `~` 项目。
2. 读取并校验 `WorkspaceSnapshot`，恢复活动项目路径；没有快照时回退现有
   `ProjectStore.activeProjectPath`，保证旧版本迁移。
3. 对匹配项目逐标签构造全新 `TerminalSession` 和 pane，再组装 `TerminalTab`。
4. 构建窗口内容，只给活动标签创建 Metal 视图；后台标签继续只持有 PTY 与 core 状态。
5. 若活动项目没有可恢复标签，沿用主队列下一轮创建首个新标签的逻辑。

恢复不会在 `TerminalCore` 引入 AppKit 或 Metal，也不会改变 cell、grid 或 scrollback 的
常驻布局。PTY 数量与恢复 pane 数一致，这是 roadmap 所要求“启动新 shell”的直接成本；
读取上限避免损坏快照把该成本放大到无界。

## 保存调度与退出顺序

主窗口新增工作区变更入口，以下成功操作后标记快照为 dirty：

- 新建、关闭、选择或重命名标签；
- 创建、关闭或激活 pane；
- 用户完成分隔线拖动并提交新比例；
- 添加、删除、重排或选择项目；
- shell 自行退出并改变标签或布局。

连续变化使用一个可取消的主队列 `DispatchWorkItem`，以约 250 ms 延迟合并为最后一份
完整快照。调度器不使用周期计时器；同一轮拖动只在现有 `onWeightsChange` 的最终提交后
触发。项目置顶、备注、颜色仍由 `ProjectStore` 即时保存，同时安排工作区快照以保持项目
顺序和活动路径一致。

`windowWillClose` 必须先取消待执行任务并同步生成、写入最后快照，然后才能解除回调、
终止 PTY 和清空 tabs。否则当前实现的清空顺序会覆盖成空工作区。对象析构只取消任务，
不再次保存。写入失败保留旧快照且不阻止退出；之后的显式变化仍可重试。

## 错误处理与隐私

- 无快照、损坏 JSON、过新 schema：忽略工作区状态并按旧项目数据启动。
- 单个项目路径不匹配：忽略该记录，不创建已被用户移除的项目。
- 单个标签布局非法或超限：丢弃该标签，继续恢复同项目其它标签。
- 工作目录缺失：按项目目录、用户主目录逐级回退。
- shell 启动失败：清理该标签已创建的会话，不保留部分布局。
- 保存编码失败：不移除上次有效数据，不打断终端操作。

快照只包含路径、用户自定义标签名、活动索引、UUID 与布局比例。它不包含终端文本、命令、
环境、进程参数或动态 OSC 标题，也不进入 iCloud 配置同步。路径本身属于本机私有应用
状态，不用于日志或遥测。

## 验证

自动验证包括：

- schema 1 JSON 的完整编解码往返；
- 损坏 JSON、过新 schema、非法轴向、权重、重复 pane ID、空分组和越深布局被拒绝；
- 项目、标签、pane 与总预算逐层生效，局部坏标签不影响其它标签；
- `ProjectStore` 旧数据在没有快照时仍能启动新标签；
- 项目关联、标签顺序与名称、活动项目/标签/pane、嵌套轴向和比例正确恢复；
- 缺失项目跳过，缺失 pane 目录回退项目目录和用户主目录；
- 快照使用前台目录，查询失败时保留初始目录；
- 连续变化只产生一次延迟保存，强制保存会取消待执行任务并立即写入；
- 窗口关闭先保存再终止并清空会话；
- 恢复后每个 pane 是新 `TerminalSession`，终端内容与 scrollback 为空；
- 完整运行 `swift test` 与 `swift build`。

运行时验收：创建多个项目、命名标签、组成横纵嵌套分屏并调整比例，在各 pane 切换不同
目录后退出 Ink；重新启动确认结构、活动位置与目录恢复，但进程、输出和历史没有恢复。
随后删除一个工作目录、破坏测试 suite 中的快照并再次启动，确认回退路径和局部降级均
不阻塞首屏。

该功能不修改渲染、grid 或 scrollback 热路径，不要求 Time Profiler 热路径采样。验收时
记录无快照与典型快照的启动耗时、快照字节数和保存耗时，确保持久化本身没有形成可见
延迟；PTY 启动成本单独按恢复 pane 数说明。
