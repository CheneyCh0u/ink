# 活跃会话关闭确认设计

## 背景

Issue #52。Ink 当前在关闭分屏、标签、项目、窗口或执行 Command-Q 时，
会直接终止对应 PTY。若 Claude Code、vim、ssh 等前台程序仍在运行，误操作
可能丢失尚未保存的工作。

本功能只保护仍有非 shell 前台程序的会话。空闲 shell 保持直接关闭，避免给
日常标签管理增加无意义的确认步骤。

## 范围

关闭确认覆盖以下入口：

- 关闭当前分屏；
- 关闭标签；
- 移除项目；
- 关闭主窗口；
- Command-Q 退出 Ink。

关闭分屏只检查目标分屏；关闭标签检查该标签的全部分屏；移除项目检查该项目的
全部标签；关闭窗口或退出 Ink 检查所有项目中的全部会话。一次用户动作最多弹出
一个确认框。

不新增设置开关，不保存“不要再询问”偏好，不修改 TerminalCore、Metal 渲染或
会话恢复行为，也不新增依赖。该功能属于 roadmap 已有的 P1「会话与窗口」范围，
无需调整 roadmap。

## 活跃程序判定

`PTYSession` 启动时记录实际配置的 shell 名称，并在用户发起关闭时读取控制终端的
前台进程组组长。`/usr/bin/login` 会 fork 实际 shell，因此不假设两者 PID 相同；
判定只与本次配置的 shell basename 比较，不依赖 `zsh`、`bash`、`fish` 等固定名称
名单，因此兼容用户配置的其他 shell。

- 前台仍是启动的 shell：会话空闲，无需确认；
- 前台进程组已切换到作业，或 shell 被 `exec` 成其他程序：需要确认；
- PTY 已退出：无需确认；
- PTY 仍在运行但系统查询失败：按安全优先处理，使用未知进程信息触发通用确认。

查询只发生在关闭动作上，不引入轮询、定时器或常驻缓存。多个关闭目标以 pane
身份去重，避免递归布局或聚合路径重复计数。

## 组件边界

### InkPTY

提供一次性前台进程快照，包含是否仍在运行、是否为启动 shell，以及可用时的前台
进程名。所有 `tcgetpgrp`、进程身份和 `sysctl` 细节留在 InkPTY，不泄漏到外壳。

### InkShell

增加纯值的关闭风险与确认内容模型。它接收目标 pane 的前台快照，负责：

- 过滤空闲 shell 与已退出会话；
- 聚合并去重需要保护的 pane；
- 生成单会话或多会话的标题、说明和破坏性按钮名称。

`MainWindowController` 负责按关闭入口收集目标 pane。在得到允许之前，不先修改
项目、标签或分屏结构，也不解除回调或终止 PTY。取消确认后，界面、焦点和进程
状态保持不变。

`AppDelegate.applicationShouldTerminate` 向主窗口控制器请求退出许可，并返回
取消或继续退出。退出许可使用一次性状态承接随后发生的窗口关闭，确保 Command-Q
不会触发第二次确认；许可不跨越本次退出流程长期保存。

## 原生确认框

使用 AppKit `NSAlert`，不自绘对话框。它继承 Ink 当前的 system、light 或 dark
appearance，自动获得 macOS 的键盘焦点、VoiceOver 和高对比度支持。取消操作支持
Escape，危险操作使用系统 destructive 样式。

单个会话的基础文案：

- 标题：`关闭正在运行的会话？`
- 说明：`claude 仍在运行。关闭后，该进程会被终止。`
- 按钮：`取消` / `关闭会话`

聚合退出的基础文案：

- 标题：`退出 Ink 并结束 3 个活跃会话？`
- 说明：`claude、vim、ssh 仍在运行。未保存的工作可能丢失。`
- 按钮：`取消` / `退出 Ink`

关闭标签、分屏或移除项目时，破坏性按钮分别使用 `关闭标签`、`关闭分屏` 和
`移除项目`。系统查询失败时不伪造进程名，改用“有会话仍在运行”的通用说明。
进程名数量过多时显示总数和有限摘要，避免对话框无限变宽或变高。

## 数据流

1. 用户触发关闭动作。
2. 窗口控制器确定该动作的 pane 范围。
3. 每个会话向 InkPTY 请求一次前台进程快照。
4. 纯逻辑模型过滤、去重并生成关闭风险摘要。
5. 若没有风险，立即执行原关闭逻辑。
6. 若有风险，显示一个原生确认框。
7. 取消时不执行任何状态变更；确认时只执行步骤 2 对应的关闭操作。

## 竞态与失败处理

前台程序可能在快照后自行退出。用户确认后，原关闭逻辑仍须容忍 PTY 已经退出，
不能因重复结束而报错。若程序在“无需确认”的快照后才启动，关闭动作与进程启动
存在不可消除的竞态；本功能以关闭时刻的内核快照为准，不增加持续监控。

确认框无法取得主窗口时，Command-Q 使用应用级原生警告；局部关闭入口只会在主
窗口存在时发生。显示确认期间忽略同一目标的重复关闭请求，防止叠加多个警告。

## 测试与验收

### 单元测试

- 实际配置的自定义 shell 被识别为空闲 shell；
- 前台进程组切换到 Claude Code、vim、ssh 等作业时触发保护；
- shell 被 `exec` 成其他程序时触发保护；
- 已退出 PTY 不触发保护，运行中但查询失败触发通用保护；
- 单 pane、标签多 pane、项目和全应用范围正确聚合与去重；
- 文案按目标类型、进程数量和未知进程正确生成；
- 取消不执行关闭闭包，确认只执行一次；
- Command-Q 的退出许可不会在窗口关闭阶段重复确认。

### 集成与人工检查

- `swift test` 全绿；
- `swift build` 零警告；
- 分别在 system、light、dark appearance 下检查原生警告的文字、按钮、焦点和
  destructive 样式；
- 实机启动空闲 shell、Claude Code、vim 和 ssh，逐一验证关闭分屏、标签、项目、
  窗口与 Command-Q；
- 验证 Escape 取消后会话继续运行，确认后只结束目标范围。

本改动不触碰 grid、scrollback 或渲染热路径，无需 Instruments 性能采样。关闭时
新增的系统调用与 pane 数量线性相关，且只执行一次，不增加常驻内存。
